Best Practices

Treat Your API Key Like a Private Key. We Certainly Do.

Follow these rules to keep your integration secure. BrillSign enforces some of these at the platform level — secure defaults are not optional.

Store in environment variables, never in source code Hardcoded keys routinely leak via git history. Use .env files or a secrets manager like AWS Secrets Manager or Vault.
Use separate keys for dev / staging / production A compromised dev key must not be able to affect production data. Scope permissions to match each environment's needs.
Rotate keys every 90 days BrillSign sends you reminder emails before expiry. Zero-downtime rotation — create new key, update your service, then revoke the old one.
Restrict permissions with scopes — least privilege by default A key used only to upload documents should not have audit:read or keys:manage scopes. Narrow scope = smaller blast radius.
Monitor usage via developer dashboard anomaly alerts BrillSign flags unusual patterns — sudden volume spikes, first-time geo access, off-hours API calls — and notifies you in real time.
Revoke instantly if compromised — zero downtime Key revocation is immediate. Issue a new key, deploy, then revoke in any order. There's no dependency between creation and revocation.

Scopes

Least Privilege by Default. Maximum Safety by Design.

Fine-grained API scopes ensure every key does exactly what it needs — nothing more. Grant only the permissions each service requires.

Zero-Knowledge

Auth Proves Who You Are. ZK Ensures No One Reads What You Sign.

Authentication and zero-knowledge are separate guarantees that work together. One identifies you. The other protects what you're doing.
