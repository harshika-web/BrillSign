Quick Integration

From Zero to Signed in 5 Lines.

Every SDK uses the same clean, predictable interface. Pick a scenario and a language — the SDK handles the rest.

upload_document.py 
import brillsign

# Initialize the client
client = brillsign.BrillSign(api_key="sk_live_...")

# Upload a document for signing
document = client.documents.upload(
    file=open("contract.pdf", "rb"),
    name="Q4 Partner Agreement",
    expires_in=7,  # days
)

print(document.id)       # doc_abc123
print(document.status)   # 'ready'
print(document.proof_hash)  # sha256:f3d9...

import brillsign

client = brillsign.BrillSign(api_key="sk_live_...")

# Verify cryptographic proof — no BrillSign account needed!

    document_id="doc_abc123",
    proof_hash="sha256:f3d9e1...",
)

Yes. All 6 SDKs are maintained by the BrillSign engineering team and receive regular updates aligned with API version releases. Python and Node.js are our tier-1 priority SDKs and receive features first. Java, Go, .NET are tier-2, and Ruby is tier-3 (community-assisted but BrillSign-maintained).

For new projects, Python or Node.js are recommended — they receive the most frequent updates, have full feature parity, and have the largest community. For enterprise Java or .NET shops, those SDKs are production-ready with full type safety. Go is excellent for high-performance services.

Yes — the verification.verify() method works with just a proof_hash and doesn't require authentication. This means your recipients can independently verify document authenticity without ever creating a BrillSign account. This is by design — cryptographic proof is yours forever.

Python, Node.js, Java, Go, and .NET SDKs all include automatic exponential-backoff retry logic for transient network errors and 429/503 rate-limit responses. Ruby currently requires manual retry handling but a built-in retry module is planned for v1.1.

Open a GitHub issue first to discuss the change. For bugs, include a minimal reproducible example. For features, explain the use case. Once aligned, fork the repo, branch from main, write tests, and open a PR. All PRs are reviewed within 5 business days by the BrillSign SDK team.
